# 项目开发历程与AI协作记录

## 基于AI的智能K歌声乐训练平台

---

# 📖 前言

本项目是深度学习与神经网络课程的大作业，旨在构建一个基于AI的智能K歌声乐训练平台。项目开发过程充满了挑战和惊喜，从技术选型、算法实现到界面优化，每一步都离不开AI工具的帮助。本文记录了项目开发的完整历程，以及与AI工具协作的精彩瞬间。

---

# 🗓️ 第一阶段：项目起步（第1周）

## 1.1 需求分析

### 初始需求
```
用户：我想要一个基于AI的K歌平台，可以实时显示音高曲线，
      对比参考音高和演唱音高，并给出AI点评。

AI：好的，我来帮您设计这个项目。
    您需要考虑以下几个方面：
    1. 音高检测算法选择（自相关/YIN/深度学习）
    2. 实时音频处理方案（Web Audio API）
    3. 数据可视化（Chart.js）
    4. AI点评系统设计
```

### 技术方案讨论
```
用户：音高检测用什么算法比较好？

AI：传统自相关算法实现简单但精度较低，
    YIN算法精度较高但计算量大，
    Swift-F0是基于深度学习的算法，精度最高。
    考虑到您的课程是深度学习，推荐使用Swift-F0。

用户：好的，那就用Swift-F0。
```

## 1.2 技术架构设计

### 架构方案
```
AI：我建议采用前后端分离架构：
    ┌──────────────────────────────────┐
    │ 前端：React + TypeScript + Vite  │
    │ 实时音高曲线 · 卡拉OK模式        │
    └──────────────────────────────────┘
                ↕ WebSocket
    ┌──────────────────────────────────┐
    │ 后端：FastAPI + Python          │
    │ WebSocket · 评分 · AI点评        │
    └──────────────────────────────────┘
                ↕
    ┌──────────────────────────────────┐
    │ AI核心：Swift-F0深度学习模型     │
    │ 高精度音高提取（误差<5音分）      │
    └──────────────────────────────────┘

用户：听起来不错，开始实现吧！
```

### 技术栈确定
| 技术 | 用途 | 原因 |
|------|------|------|
| React 18 + TypeScript | 前端框架 | 类型安全，生态成熟 |
| Vite | 构建工具 | 极速HMR |
| Chart.js | 数据可视化 | 轻量级，支持实时数据流 |
| Web Audio API | 音频处理 | 浏览器原生支持 |
| FastAPI | 后端框架 | 高性能，自动文档 |
| Python 3.11 | 后端语言 | AI/ML生态丰富 |
| Swift-F0 | 音高检测 | 深度学习驱动，精度高 |
| WebSocket | 实时通信 | 全双工，低延迟 |

# 🔧 第二阶段：核心功能开发（第2-3周）

## 2.1 前端音高曲线显示问题

### 问题出现
```
用户：为什么前端音高曲线没有显示？

AI：让我检查一下代码...
    发现问题了！CSS布局冲突，.content-grid重复定义，
    而且Chart.js的CDN被浏览器拦截了。

用户：怎么修复？

AI：1. 改用本地Chart.js资源
    2. 重构CSS布局，确保容器尺寸正确
```

### 修复过程
```
AI：我来帮您修改代码。
    将CDN链接改为本地路径：
    frontend/node_modules/chart.js/dist/chart.umd.min.js

用户：好的，我刷新页面试试...还是没显示！

AI：请检查浏览器控制台日志...
    看到了！Chart.js加载失败，路径不对。
    需要调整路径指向正确的位置。

用户：好的，修复了！曲线显示出来了！
```

## 2.2 时间对齐问题

### 问题描述
```
用户：参考音高和演唱音高的时间戳不同步，导致曲线错位。

AI：这是因为录制和演唱的起始时间不同。
    需要实现时间对齐容差调整功能，
    通过滑动窗口匹配最近的参考音高。
```

### 解决方案
```
AI：我来实现时间对齐逻辑：
    1. 使用audioContext.currentTime替代Date.now()
    2. 实现±0.3秒的时间容差匹配
    3. 添加时间对齐容差调整滑块

用户：好的，试试看...对齐效果好多了！
```

## 2.3 音高偏差过大问题

### 问题分析
```
用户：为什么音分偏差会达到-681.94？这太离谱了！

AI：让我分析一下数据...
    发现问题了！前端自相关算法对噪声敏感，
    产生了大量异常值，比如923Hz、1000Hz等超出人声范围的值。
```

### 优化方案
```
AI：我来优化自相关算法：
    1. 添加噪声门限过滤（NOISE_THRESHOLD = 0.005）
    2. 添加置信度检测（CONFIDENCE_THRESHOLD = 0.3）
    3. 添加历史平滑（HISTORY_SIZE = 5）
    4. 增加亚采样精度优化和中值滤波

用户：效果好多了，异常值少了很多！
```

# 🤝 第三阶段：AI协作突破（第4周）

## 3.1 前后端算法不一致问题

### 核心矛盾
```
用户：前后端使用不同的音高提取算法，导致曲线不重合！

AI：这确实是个大问题。前端用自相关，后端用Swift-F0，
    算法差异必然导致结果不一致。

用户：怎么解决？

AI：方案有两种：
    方案A：前端也集成Swift-F0（复杂，需要WASM）
    方案B：前端将音频数据发送到后端，
           由后端统一使用Swift-F0提取音高

用户：选方案B，比较简单。
```

### 实现过程
```
AI：我来实现WebSocket通信：
    1. 前端通过WebSocket发送音频数据到后端
    2. 后端使用Swift-F0提取音高
    3. 后端返回音高结果给前端

用户：好的，开始实现...遇到问题了！
    WebSocket连接老是断开！

AI：让我检查一下代码...
    发现了！useEffect依赖数组配置有问题，
    导致组件每次渲染都重新连接。
    我来修复这个问题。
```

## 3.2 Swift-F0集成问题

### 版本冲突
```
用户：安装Swift-F0时报错了！

AI：让我看看错误信息...
    是Python版本和依赖包版本不兼容。
    需要安装指定版本的依赖：
    numpy==1.26.4, torch==2.2.0等

用户：好的，安装成功了！
```

### 采样率问题
```
用户：为什么参考音高和演唱音高的曲线还是不对齐？

AI：我来分析一下数据...
    发现了！前端使用48000Hz采样率，
    后端Swift-F0使用16000Hz采样率，
    采样率不一致导致时间计算错误！

用户：怎么修复？

AI：将前端采样率统一改为16000Hz，
    这样前后端就一致了。
```

### 时间戳对齐问题
```
用户：演唱模式下参考音高显示为一条直线！

AI：让我检查后端代码...
    发现了！后端用帧序号直接索引参考音高数组，
    但录制和演唱的帧计数不一致。
    需要改用前端发送的时间戳查找最近的参考音高。

用户：修复了！曲线对齐了！
```

### 文档生成

### 创意激发

## 6.2 与AI协作的最佳实践

### 清晰的需求描述
```
用户：我想要一个实时音高对比功能。
✅ 好的描述

❌ 模糊的描述
```

### 提供上下文
```
用户：音高曲线显示有问题，我用的是Chart.js，
      错误信息是"Cannot read property 'getContext' of null"。

AI：根据上下文，问题可能是canvas元素还没加载就调用了getContext。
```

### 逐步迭代
```
不要一次提出太多需求，逐步迭代开发。
例如：
1. 先实现基础功能
2. 再优化细节
3. 最后完善体验
```

### 验证和测试
```
AI生成的代码需要验证和测试，不能直接使用。
例如：
- 运行测试用例
- 手动测试功能
- 检查代码质量
```

## 6.3 AI工具的局限性

### 代码理解深度有限
```
AI可能无法完全理解复杂的业务逻辑，
需要用户提供足够的上下文和解释。
```

### 依赖环境差异
```
AI可能不了解本地环境的具体配置，
需要用户自己调整依赖和配置。
```

### 创意受限
```
AI的创意来源于训练数据，
可能无法提出真正突破性的创新方案。
```
